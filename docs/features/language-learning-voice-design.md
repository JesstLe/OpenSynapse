# OpenSynapse 语言学习与语音交互设计

**项目**: OpenSynapse  
**日期**: 2026-03-28  
**适用范围**: 英语 / 口语 / 语言学习产品设计、语音交互、阶段化学习算法、UI 结构

---

## 1. 目标

我们希望 OpenSynapse 后续支持：

1. 用户学习英语或其他语言
2. 重点覆盖口语、听力、表达、纠错和长期掌握
3. 支持文本与语音混合交互
4. 从基础到进阶再到“接近自动化输出”的全过程
5. 与现有：
   - AI 会话
   - 笔记提炼
   - FSRS 复习
   - 知识图谱
   
形成统一闭环

一句话：

**这不是单纯做一个“AI 口语陪练”，而是做一个从输入、练习、纠错、强化到掌握的语言训练系统。**

---

## 2. 产品定位

建议把语言学习能力定义为：

**OpenSynapse 的一个新学习域（Learning Domain），而不是一套完全分裂的新 App。**

理由：

1. 当前 OpenSynapse 已经有学习对话、知识提炼、主动召回和知识库
2. 语言学习本质上同样需要：
   - 对话练习
   - 结构化知识点
   - 间隔重复
   - 进度追踪
3. 如果完全单独做一套侧边栏，容易造成：
   - 用户资产分裂
   - 会话与知识库脱节
   - 实现复杂度陡增

## 3. 入口设计：单独侧边栏还是放进现有结构

## 3.1 我的建议

**第一阶段不要做第二套侧边栏。**

更推荐：

- 在现有左侧导航中新增一个一级入口：`语言训练`

位置建议：

- 仪表盘
- 学习对话
- 语言训练
- 知识图谱
- 主动召回
- 知识库
- 设置

原因：

1. 语言学习是重要场景，值得一级入口
2. 但它仍是 OpenSynapse 的一个学习域，不该在第一版就完全独立成第二套应用
3. 未来如果语言模块变得足够强，再在 `语言训练` 内部做自己的二级导航即可

## 3.2 二级导航建议

进入 `语言训练` 后，内部推荐 4 个标签：

1. `会话`
2. `口语`
3. `课程`
4. `复习`

含义：

- `会话`
  日常语言聊天、情景对话、自由表达

- `口语`
  重点做语音输入、发音纠错、跟读、口语评分

- `课程`
  系统化单元，例如：
  - 零基础发音
  - 日常英语
  - 商务英语
  - 面试英语
  - 出国场景英语

- `复习`
  专门展示语言学习中的词汇、句型、易错点、发音最小对立对等

## 3.3 为什么不是“单独再开一套完整侧边栏”

如果一开始就做完全独立侧边栏，短期会有几个问题：

1. 导航复杂，像两个产品拼在一起
2. 知识库、复习系统和语言模块的数据归属不清晰
3. 后续要做“语言知识点进入主知识库”时会更难

所以更合理的演化路线是：

- 第一阶段：一个一级入口 `语言训练`
- 第二阶段：在其内部做二级导航与专用工作台
- 第三阶段：如果 DAU 和内容规模足够，再考虑它是否成长为近似独立子产品

## 4. 用户最终应该获得什么

语言学习不是“和 AI 聊天聊多了就会”，它至少要覆盖 5 个层次：

1. **识别**
   看见 / 听见后能理解

2. **回忆**
   不看答案时能想起来

3. **生成**
   能自己说出来 / 写出来

4. **纠错**
   能发现哪里不自然、哪里错误

5. **自动化**
   在真实场景里流畅使用，不再依赖慢速翻译式思考

OpenSynapse 的价值应该是帮助用户跨越这 5 层，而不是只停在“背单词”。

## 5. 核心产品能力设计

## 5.1 会话式语言导师

这是语言模块的基础层。

能力包括：

- 自由对话
- 场景扮演
- 难度自适应
- 实时纠错
- 术语解释
- 中英切换解释

导师人格建议至少支持：

- 零基础陪练教练
- 发音教练
- 商务英语教练
- 考试英语教练
- 旅行场景教练

## 5.2 语音交互

语音能力建议拆成三段：

1. **语音输入**
   - 用户说话
   - STT 转文本

2. **口语分析**
   - 识别词汇错误
   - 识别语法错误
   - 识别不自然表达
   - 识别发音偏差

3. **语音输出**
   - AI 朗读标准表达
   - 跟读模式
   - 对比模式

这意味着后续技术上要接入：

- STT（speech-to-text）
- TTS（text-to-speech）
- 语音评估 / 发音评分

## 5.3 从会话自动提炼语言资产

语言对话结束后，系统不应只停在“聊过了”。

应自动提炼出：

- 新词汇
- 高频句型
- 用户错误句
- 推荐替换表达
- 发音难点
- 场景模板

并进入：

- 语言知识库
- 语言复习卡

## 5.4 多模态输入

适合语言学习的输入包括：

- 文本
- 语音
- 图片描述
- 场景卡片
- 真实材料（文章、字幕、PDF、网页）

这样用户可以做：

- 看图说话
- 听音复述
- 材料精读
- 口头总结

## 6. 学习算法：如何从基础到进阶再到完全掌握

这个部分是核心。

我建议不要只靠一个算法，而是做一个 **四层学习引擎**：

1. **能力分层模型**
2. **知识点掌握模型**
3. **记忆调度模型**
4. **输出自动化模型**

---

## 7. 第一层：能力分层模型

语言学习的阶段，不应只按“学了多少课”判断，而要按能力层级判断。

建议参考 CEFR 思路，但做成 OpenSynapse 自己的内部分层：

### Stage 0：启蒙

特点：

- 只能理解极少量高频表达
- 依赖中文解释
- 无法独立开口

### Stage 1：生存表达

特点：

- 能完成简单自我介绍、购物、问路、点餐
- 可以做极短轮次对话
- 错误多，但有基本表达意图

### Stage 2：场景沟通

特点：

- 能在熟悉场景持续交流
- 可以描述经历、观点、计划
- 仍存在明显母语迁移痕迹

### Stage 3：结构化表达

特点：

- 能进行较长说明
- 能组织逻辑
- 能区分正式 / 非正式表达

### Stage 4：流利与准确并进

特点：

- 反应更快
- 错误率明显下降
- 能适配不同语境

### Stage 5：自动化掌握

特点：

- 不再需要大量内部翻译
- 在真实场景中稳定输出
- 具备迁移能力和纠错能力

---

## 8. 第二层：知识点掌握模型

语言学习不是只有单词，还包括很多不同粒度的知识对象。

建议统一抽象成 `LanguageSkillUnit`：

```ts
type LanguageSkillUnit =
  | VocabularyUnit
  | PhraseUnit
  | GrammarPatternUnit
  | PronunciationUnit
  | SituationTemplateUnit
  | ErrorPatternUnit;
```

每个 unit 应至少包含：

- `skillType`
- `targetLanguage`
- `difficulty`
- `comprehensionScore`
- `recallScore`
- `productionScore`
- `pronunciationScore`
- `fluencyScore`
- `errorCount`
- `lastSeenAt`
- `lastUsedAt`

### 为什么不能只用一个“掌握度”

因为语言里“看得懂”和“说得出来”完全不是一回事。

比如一个单词：

- 你能认出来
- 你能听懂
- 但你未必能自己说出来
- 更未必能在句子里自然用出来

所以建议至少拆成：

1. `recognition`
2. `recall`
3. `production`
4. `pronunciation`

## 9. 第三层：记忆调度模型

这里可以直接复用你们现有的 FSRS 思路，但要“语言化”。

当前项目已有：

- [src/services/fsrs.ts](/Users/lv/Workspace/OpenSynapse/src/services/fsrs.ts)

这是一个很好的基础，但语言学习里不应该只拿它管“单张卡片”。

### 9.1 推荐做法

FSRS 继续负责：

- 词汇
- 句型
- 易错点
- 发音对立对

也就是：

**对记忆型对象做最优复习间隔调度。**

### 9.2 语言卡片的 rating 不应只看“对 / 错”

建议把语言卡片复习结果映射成：

- Again：完全不会
- Hard：知道但输出困难
- Good：基本可以正确回忆与使用
- Easy：可以快速自然使用

这样会比普通知识卡更贴近语言掌握。

### 9.3 发音卡要有单独调度维度

因为发音的遗忘曲线和词义记忆不完全一样。  
发音建议单独作为 `PronunciationUnit` 卡组处理。

## 10. 第四层：输出自动化模型

“完全掌握”不是记住，而是自动化。

这里建议增加一个 **Output Automaticity Score**。

它不只是看用户会不会，而是看：

1. 反应时间
2. 是否依赖提示
3. 是否反复修正
4. 是否需要中文中介
5. 能否在新场景迁移使用

### 10.1 自动化评分维度

建议每次口语 / 写作任务后记录：

- `latencyMs`
- `selfRepairCount`
- `promptDependency`
- `grammarErrorDensity`
- `lexicalRange`
- `taskCompletion`

然后生成一个 `automaticityScore`。

### 10.2 为什么这很重要

因为从“会”到“真掌握”，中间差的正是自动化。

如果没有这一层，系统只会帮助用户：

- 记住
- 理解

但很难帮助用户：

- 流畅使用

## 11. 推荐的完整学习引擎

最终建议的算法框架：

### 11.1 阶段分级：能力分层

用阶段模型判断用户整体水平。

### 11.2 微观掌握：多维 mastery score

每个语言 unit 有多维掌握分：

- 识别
- 回忆
- 生成
- 发音
- 自动化

### 11.3 记忆调度：FSRS

用 FSRS 管理复习节奏。

### 11.4 难度自适应：Mastery-based progression

根据 mastery score 决定：

- 下一个练习难度
- 是否进入下一阶段

### 11.5 场景泛化：Curriculum graph

把语言知识组织成图谱：

- 词汇
- 句型
- 场景
- 语法
- 发音

在用户完成一个阶段后，系统自动扩展到相邻技能。

目标：

- 在新场景中流畅使用

方法：

## 13. 语音交互的核心设计

## 13.1 交互模式

推荐支持三种语音模式：

### 模式 A：按住说

适合：

- 初期 MVP
- 简单稳定

### 模式 B：连续对话

适合：

- 口语陪练
- 实时交流

### 模式 C：跟读 / 对比

适合：

- 发音训练
- 语调模仿

## 13.2 语音评估不只给一个总分

应该拆成：

- 发音准确度
- 音节重音
- 连读与节奏
- 流畅度
- 可理解度

并给出：

- 最少的关键纠错点
- 标准读法示范
- 跟读重试

## 13.3 不要让语音反馈太啰嗦

口语练习时，每轮反馈应该非常克制：

- 先鼓励输出
- 再指出最关键 1-2 个问题
- 再给更自然版本

而不是每句都做论文式批注。

## 17. 产品层面的最终建议

如果现在问我最推荐的形态，我的答案是：

### 17.1 UI 结构

第一阶段：

- **新增一个一级入口：`语言训练`**

不要一开始就做第二套独立侧边栏。

### 17.2 学习算法

不要只用一个算法。

最佳组合是：

- **能力分级模型**：决定用户在什么阶段
- **多维 mastery score**：决定某个词 / 句型 / 发音掌握到什么程度
- **FSRS**：决定什么时候复习
- **自动化评分**：决定是否真的从“会”进化到“自然使用”**

### 17.3 价值点

OpenSynapse 语言模块的真正差异化，不该只是：

- “能和 AI 用英语聊天”

而应该是：

- “能把一次语言对话自动转成长期掌握系统”

也就是：

**会话练习 + 纠错 + 资产提炼 + 复习调度 + 阶段升级**

---

## 18. 一句话结论

OpenSynapse 后续做语言学习，最合理的路线是：

- 产品上新增 `语言训练` 一级入口
- 内部再分 `会话 / 口语 / 课程 / 复习`
- 算法上采用：
  - 阶段分级
  - 多维 mastery score
  - FSRS 复习调度
  - 自动化评分

这样才能真正支持用户从：

**基础 -> 进阶 -> 熟练 -> 自动化掌握**

而不是停留在“和 AI 聊了很多句子”的层面。
